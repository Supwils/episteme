## 一、总判断

这个仓库已经不是早期脚手架，而是一个内容体量大、路由覆盖广、工程质量门基本可靠的统一知识平台。当前主线状态可以概括为：

- **可运行**：`pnpm install`、`typecheck`、`lint`、`test`、`build`、`bundle-check` 均通过。
- **内容强**：多数领域不是百科摘要，已经有较多本科到研究生级长文。
- **产品方向需要重新收紧**：书签、阅读历史、分享、主题切换等功能与 `docs/聚焦方向.md` 明确“不做”的边界冲突。
- **最大价值漏损是“写好了但不可达/错达”**：搜索、知识库路由、跨域链接和内容源分叉仍会让已有内容无法被普通读者稳定发现。
- **长期维护风险在架构卫生**：内容真相源分裂、loader 重复、领域注册点分散、共享包边界偏软、大型可视化组件仍多。

建议的总体优先级：

1. 先清理产品边界冲突和搜索/路由 404。
2. 再统一内容真相源与 loader。
3. 然后补齐 CI 发布门和可访问性基线。
4. 最后做内容深度补强和大文件拆分。

## 三、P0 / P1 关键发现

### P0：产品边界冲突仍然存在

`docs/聚焦方向.md` 明确不做阅读进度持久化、收藏夹、分享、暗/亮模式切换。但当前全局产品面仍存在：

- `apps/portal/components/ReadingPanel.tsx`
- `apps/portal/lib/bookmarks.ts`
- `apps/portal/lib/reading-progress.ts`
- `apps/portal/components/DailyShareCard.tsx`
- `apps/portal/components/ThemeToggle.tsx`

影响：

- 文档声称越界功能已删除，但代码仍有书签/阅读历史/分享/主题切换入口，形成产品与文档漂移。
- 后续如果继续扩展这些功能，会偏离“知识内容平台”的核心定位。

建议：

- 默认移除或隐藏这些入口。
- 如果产品决定保留，必须先修改 `docs/聚焦方向.md`，不能让“真相源”与代码相互矛盾。

### P1：搜索与路由可达性断裂

已发现的高价值修复点：

- `apps/portal/components/GlobalSearch.tsx` 的结果分组遗漏 `mathematics` 和 `cosmology` 展示。
- `apps/portal/lib/search-index/psychology-index.ts` 会生成 `/psychology/knowledge-base/[slug]`，但 `apps/portal/app/psychology/knowledge-base` 路由不存在。
- `apps/portal/lib/search-index/cosmology-index.ts` 会生成 `/cosmology/dialogues/[slug]`，但 `apps/portal/app/cosmology/dialogues` 路由不存在。
- `content/universe-physics/knowledge-base` 和 `content/universe-physics/dialogues` 有内容，但 `apps/portal/app/universe-physics` 下没有对应知识库/对话路由。
- `content/mathematics/knowledge-base` 有 23 篇 MD，但 `apps/portal/app/mathematics` 下没有 `knowledge-base` 路由。

建议优先顺序：

1. 修全局搜索分组与错误 section。
2. 对不存在的搜索结果要么补路由，要么停止索引。
3. 为物理、数学、心理学、宇宙学对话补统一 knowledge/dialogue 路由。
4. 增加搜索结果 URL 可达性测试。

### P1：内容源分裂

当前存在两套内容位置：

- 根 `content/`
- `apps/portal/content/`

风险示例：

- `@/content` 在 portal 内解析到 `apps/portal/content`。
- `apps/portal/lib/content-paths.ts` 的 generic loader 指向根 `content`。
- `content/universe-physics/cosmos/T0.ts` 与 `apps/portal/content/universe-physics/cosmos/T0.ts` 不同，较长内容可能没有上线。

建议：

- 明确根 `content/` 为唯一知识内容真相源。
- `apps/portal/content/` 只保留无法迁移的 app 内静态索引，或逐步清空为兼容层。
- 建立 drift-check，阻止同名内容两边继续分叉。

### P1：质量门还不是完整发布门

当前 CI 已跑：

- build
- typecheck
- lint
- unit test
- bundle-check

缺口：

- Playwright e2e 未进入 CI。
- `pnpm check-content` 未进入 CI，且 warning 不失败。
- 没有安全头 smoke test。
- 没有 sitemap/动态路由可达性抽样。
- packages 的 lint 多为占位脚本。
- 没有依赖漏洞扫描。

建议：

1. 先把现有 e2e 接入 CI。
2. 将 `check-content` 拆成 error 必失败、warning 可配置。
3. 加安全头 smoke test。
4. 给 `packages/ui`、`packages/content`、`packages/graph-engine` 加真实 lint/test。

## 四、内容质量结论

### 内容不是空壳

子代理抽样和本地统计一致：经济学、心理学、生命科学、宇宙学、宇宙物理、数学的大量文章都有真实数据、理论解释、案例、争议和参考文献。项目的内容底子明显强于普通百科摘要。

### 但内容深度不均匀

薄文件问题主要集中在：

- `content/philosophy/thinkers` 中 28 个新增人物页低于 80 行，典型如 `putnam.mdx`、`kripke.mdx`、`chalmers.mdx`。
- `content/human-history/knowledge-base/事件/甲午战争.md` 约 67 行，低于 KB 80 行标准。
- `apps/portal/content/philosophy/dialogues/*.mdx` 多数只有 20 多行，像残留或图谱补丁内容。
- 经济学仍有一批低于 120 行的核心 MDX，需要补强。

建议：

- 把“补了页面”与“达教授级讲稿”分开验收。
- 对 28 个哲学人物页补原典文本、思想争议、后世解释、跨领域连接、参考文献。
- 对核心科学/经济/心理文章补 DOI、URL、机构数据源，而不仅是书目式参考。

## 五、架构与可扩展性

架构健康度估计：**7.1 / 10**。

健康部分：

- 单一活跃应用 `apps/portal`。
- 领域代码按 `src-*` 分区。
- `@universe/graph-engine` 已下沉，主干跨域 import 基本清理。
- TypeScript strict 与 eslint no-any 已有实际约束。

主要风险：

- 内容真相源分裂。
- MDX/content loader 重复实现。
- 新增领域需要改多个注册点：导航、搜索、图谱、cross-link、sitemap、eslint domain、首页统计。
- 共享包 `packages/*` 仍像源码目录，`main/exports` 指向 `src/index.ts`，包内 lint/test 较弱。
- 历史板块仍有 JS/命令式渲染路径作为活跃代码。
- 大文件仍系统性存在，典型：
  - `apps/portal/src-mathematics/components/visualizations/GeometryExplorer.tsx`：1043 行。
  - `apps/portal/src-philosophy/components/visualizations/ThoughtExperimentTree.tsx`：953 行。
  - `apps/portal/src-psychology/components/visualizations/BiasTaxonomy.tsx`：856 行。
  - `apps/portal/src-cosmology/components/visualizations/HRDiagram.tsx`：819 行。

建议重构路线：

1. 统一内容源。
2. 抽统一 content loader。
3. 建一个领域 registry，驱动 nav/search/sitemap/cross-link/graph adapter。
4. 强化 packages 的 lint/test/build。
5. 分领域拆大文件，每次只拆一组高风险可视化。

## 六、前端体验与可访问性

优势：

- 全局 `MotionProvider` 已尊重用户 reduced motion。
- 知识图谱是最成熟的 Canvas 可访问性范例，有键盘说明、详情面板、移动端控件。
- 宇宙物理 R3F 有 DPR、antialias、性能监控和降级意识。
- bundle 预算已机器化。

缺口：

- 历史 Atlas Canvas 只有 `tabIndex`，缺少 `role`、`aria-label`、键盘说明和焦点管理。
- Atlas 面包屑使用可点击 `span`，语义弱。
- 数学可视化 reduced-motion 覆盖最弱，`src-mathematics` 几乎没有专门守卫。
- `GravitySimulation` 读取 reduced motion，但 RAF 逻辑未实际使用该值。
- `packages/graph-engine` renderer 即使 dirty=false 仍持续 RAF，空闲耗电。

建议：

1. 把知识图谱的 Canvas 可访问性模式抽成 checklist，优先套到 Atlas、数学几何、生命树、宇宙学模拟。
2. 为数学全域补 reduced-motion 分支。
3. 修复 GraphRenderer 空闲持续 RAF。
4. 移动端人工验证 `/knowledge-graph`、`/human-history/atlas`、`/universe-physics/universe/*`、数学几何/概率模拟。

## 七、部署与安全

已改善：

- admin 后台和 admin API 当前不存在。
- API 仅剩 `/api/daily` 和 `/api/og`。
- 基础安全头已集中在 `apps/portal/next.config.ts`。

仍需专项：

- 生产 CSP 仍有 `script-src 'unsafe-inline'`。
- `code.iconify.design` 是运行时第三方脚本供应链点。
- `/api/og` 对 query 文本缺少长度限制，存在轻量资源滥用风险。
- 根 `vercel.json` 与 `apps/portal/vercel.json` 对 Root Directory 假设不一致。
- `human-history` 与 `universe-physics` 是 gitlink 但无 `.gitmodules`，fresh clone 语义不稳定。

建议：

- Vercel 配置需要先确认项目 Root Directory，再合并单一事实源。
- CSP nonce 迁移作为安全专项，不要夹在普通功能任务里。
- 由仓库所有者决定旧 gitlink 是正式 submodule 还是转普通目录。
