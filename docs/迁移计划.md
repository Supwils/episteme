## 迁移步骤（有序）

### 步骤 1 — 创建 @universe/ui 共享包

**为什么先做这个**：`apps/universe-physics/package.json` 和 `apps/philosophy/package.json` 都声明了 `"@universe/ui": "*"` 依赖。不创建这个包，`pnpm install` 会报错，整个工作区无法运行。

**操作**：
```bash
mkdir -p packages/ui/src
# 创建 packages/ui/package.json（name: @universe/ui）
# 创建初始组件（Button, Panel 等）
# 从根目录运行 pnpm install 链接工作区
```

**完成标准**：`pnpm install`（根目录）无报错，`@universe/ui` 可被 apps 引用。

**风险**：无重大风险，纯新增。

### 步骤 2 — universe-physics：源码迁移

**操作**：
```bash
# 将旧位置源码复制到 apps/
cp -r universe-physics/. apps/universe-physics/

# 处理 package.json 冲突：
# apps/universe-physics/package.json 已存在（工作区版本），保留它
# 复制来的 package.json 仅参考 scripts 和 pnpm.onlyBuiltDependencies

# 从根目录安装和验证
pnpm install
pnpm --filter @universe/universe-physics dev
```

**验证路径**：
1. `http://localhost:3000`（或 3033）能加载首页
2. 从 `/universe/observable` 下钻到 `/universe/solar-system/earth` 无报错
3. `pnpm --filter @universe/universe-physics build` 通过

**完成后**：旧位置 `universe-physics/` **暂不删除**，等新版上线生产并验证稳定后再删。

**关键注意事项**：

| 项目 | 旧版本 | 新版本 | 处理方式 |
|------|--------|--------|---------|
| 开发端口 | `-p 3033`（`start` 脚本） | 无显式端口（`dev` 脚本） | 保留 `-p 3033` 到 start，dev 端口冲突时在 turbo.json 配置 |
| `pnpm.onlyBuiltDependencies` | 在 package.json | 需确认在 apps/ 版本中存在 | 复制时检查是否丢失 |
| Node.js 版本要求 | ≥22.0.0 | 与旧版一致 | pnpm-workspace.yaml 或 .nvmrc 统一 |

### 步骤 3 — human-history：Next.js 迁移

**这是最复杂的迁移**。human-history 目前是 Vite 8 + 原生 JavaScript SPA，需要迁移到 Next.js 15 + TypeScript。

**迁移策略**：保留内容，重写框架层。

#### 3A — 内容层迁移（可先做）

知识库内容（`human-history/docs/`）是纯 Markdown，与框架无关，可以直接复用或保持原位引用。

数据层（`human-history/website/src/data/`）需要：
1. 将 `scholarly-details.js`、`index.js` 等转为 TypeScript（或先保留 JS 在 .ts 文件中逐步迁移）
2. 添加 Zod schema 校验数据格式
3. 确保 80+ 深度讲稿内容完整迁移

#### 3B — 路由层迁移

| 旧 Vite 路由 | 新 Next.js App Router |
|-------------|----------------------|
| `/` | `app/page.tsx` |
| `/timeline` | `app/timeline/page.tsx` |
| `/atlas` | `app/atlas/page.tsx` |
| `/graph` | `app/graph/page.tsx` |
| `/map` | `app/map/page.tsx` |
| `/figures` | `app/figures/page.tsx` |

SPA fallback（`historyApiFallback`）由 Next.js App Router 天然处理，无需额外配置。

#### 3C — 组件层迁移

| 原生 JS 模块 | Next.js 处理方式 |
|------------|---------------|
| `src/pages/*.js`（渲染函数） | `app/*/page.tsx`（Server/Client Components） |
| `src/lib/dom.js`（DOM 工具） | 保留为 `use client` 工具函数 |
| `src/lib/router.js`（自研路由） | 删除，由 Next.js Router 替代 |
| GSAP 动画 | 包裹在 `'use client'` 组件中，用 `useEffect` 调用 |
| Canvas 动画（atlas.js、graph.js） | `'use client'` + `useEffect` + `useRef`，清理逻辑不变 |

#### 3D — 高风险点

| 风险 | 具体说明 | 缓解方案 |
|------|---------|---------|
| GSAP 服务端执行 | `registerPlugin` 和 `gsap.from()` 在 SSR 报错 | 所有 GSAP 代码包裹在 `'use client'` + `useEffect` |
| Canvas 初始化时序 | Canvas 依赖 DOM ready，SSR 中 canvas 不存在 | `'use client'` + `useRef` + `useEffect(() => {}, [])` |
| `advancedChunks` 分包 | Vite `advancedChunks` 无 Next.js 等价物 | 用 `next/dynamic` 或 App Router 路由自动分包替代；构建后对比 bundle size |
| 深度讲稿数据量大 | scholarly-details.js 约 186KB gzip | 考虑按时期/域分文件 + `dynamic import`，避免首屏全量加载 |
| CSS 变量系统 | 原生 CSS custom properties（设计令牌） | 可直接迁移到 globals.css，与 Tailwind v4 兼容 |

**完成标准**：
- `pnpm --filter @universe/human-history build` 无错误
- `/atlas`、`/graph` Canvas 页面在桌面和移动端正常渲染
- 所有 80+ 深度讲稿内容可访问
- 页面切换无内存泄漏（验证 Canvas 清理）

**完成后**：旧 Vite 版本 `human-history/website/` **暂不删除**，等新版生产验证后再删。

### 步骤 4 — portal：实现导航门户

**前提**：步骤 1 完成（@universe/ui 存在）。

**功能范围**：
- 平台首页：展示三大知识领域（宇宙物理、人类历史、哲学思想）
- 各应用入口卡片，视觉预览
- 导航到各子应用的链接
- 可选：跨领域推荐（如从哲学家跳转到对应历史时期）

**技术要求**：
- Server Components 优先（无需 `use client` 的地方不用）
- 响应式：Desktop + Mobile 都要好看
- 首屏加载 < 1s（LCP）

## 风险登记册

| 风险 | 概率 | 影响 | 缓解方案 |
|------|------|------|---------|
| @universe/ui 接口设计不稳定，导致 apps 频繁调整 | 中 | 中 | 先实现最小接口（Button、Panel），不要过度设计 |
| human-history Canvas 在 Next.js 中初始化报错 | 高 | 高 | 早期搭建 PoC 验证 atlas.js 迁移路径 |
| Vite `advancedChunks` 迁移后 bundle 变大 | 中 | 中 | 迁移后立刻用 `pnpm bundle-check` 对比 |
| pnpm-workspace.yaml 缺失 | 低 | 高 | 立即检查并创建 |
| universe-physics 旧版 npm 包与 pnpm 工作区冲突 | 低 | 中 | universe-physics 旧版保持 npm，apps 版本用 pnpm |
| human-history 80+ 讲稿数据迁移出错/丢失 | 低 | 极高 | 迁移前备份，迁移后用 lint:content 脚本验证 |
