# 物理 3D 场景性能预算

## 目标与边界

宇宙物理的沉浸式路线包含 Three.js、React Three Fiber、后处理和八个实时场景，不能套用通用文章的 220 KB JavaScript 预算。本预算按用户实际经历拆成四层：

1. **导航外壳**：路由、WebGL 探测、载入状态和无 WebGL 降级页。
2. **WebGL 运行时**：Three.js、R3F、相机、HUD、交互和渲染器。
3. **首个场景**：由 URL 直接决定的 T0-T7 场景，不得先下载默认 T0。
4. **后续增强**：首场景完成后加载后处理；桌面空闲时只预热一个相邻场景，移动端、低性能设备、省流和 2G 网络不预热。

手写 SVG 路线有独立场景和 HUD，目前约 480 KB gzip。它不再加载 WebGL 运行时，但仍需后续按相同方法拆分手写外壳、首图和后续图。

## 依赖清单

| 层级 | 主要依赖 | 进入时机 |
| --- | --- | --- |
| 导航外壳 | Next.js 路由、WebGL 探测、载入与降级组件 | SSG 首屏 |
| WebGL 运行时 | `three`、`@react-three/fiber`、相机、HUD、Zustand、GSAP | WebGL 探测成功后 |
| 首个场景 | `ActiveScene` 对应的 T0-T7 单个动态模块 | 运行时挂载后，由 URL 直接选择 |
| 后处理 | `@react-three/postprocessing`、`postprocessing` | 首场景提交后，在浏览器空闲期 |
| 相邻场景 | 当前层级的下一个场景，T7 使用 T6 | 仅桌面/中高质量且非省流网络的空闲期 |

## 静态预算

`pnpm audit:physics-runtime`读取生产构建 manifest 并验证：

- 3D 路由初始 JavaScript gzip 不超过 180 KB。
- 轻量外壳不能静态引入 R3F 或 Three.js。
- WebGL 条件入口必须使用 `React.lazy`，不能使用会生成预取提示的 `next/dynamic`。
- T0-T7 必须保持八个独立动态导入。
- 后处理必须保持延迟加载。

当前生产构建中，3D 路由 `First Load JS` 约 149 KB。WebGL 运行时、场景和后处理的完整进入成本由下述浏览器网络预算统一约束，避免用 manifest 可达图代替实际请求。

## 运行时预算

`pnpm test:e2e:physics`使用生产模式和系统 Chrome，分别执行桌面与移动端验收。固定测试入口为地球深链接，以覆盖纹理、着色器、后处理和相邻场景预热。

| 指标 | 桌面预算 | 移动预算 |
| --- | ---: | ---: |
| 全部已加载 JS | ≤ 920,000 bytes | ≤ 920,000 bytes |
| JS heap | ≤ 64 MiB | ≤ 56 MiB |
| 首场景就绪 | ≤ 5,000 ms | ≤ 6,000 ms |
| 像素比 | ≤ 2 | ≤ 1 |
| 几何体 | ≤ 40 | ≤ 32 |
| 纹理 | ≤ 36 | ≤ 20 |
| 自动化环境 FPS | ≥ 30 | ≥ 30 |
| 非黑像素比例 | ≥ 10% | ≥ 10% |
| 600 ms 帧差均值 | ≥ 0.25 | ≥ 0.25 |

几何体和纹理是浏览器可获得的 GPU 资源代理，不等同于显存字节。后处理会重置Three.js的逐帧渲染统计，因此不把最后一个pass的draw call误作整帧预算。FPS 是系统 Chrome 自动化环境的回归底线，不代表所有终端的体验承诺；真实设备仍应覆盖集成显卡、主流手机和省电模式。

## 实测基线

2026-07-16 本地系统 Chrome 生产模式：

| 指标 | 桌面 | 移动 |
| --- | ---: | ---: |
| 全部已加载 JS | 895,119 bytes | 888,687 bytes |
| JS heap | 14.6 MiB | 13.6 MiB |
| 首场景就绪 | 793 ms | 786 ms |
| 像素比 | 1 | 1 |
| FPS | 106.1 | 106.2 |
| 几何体 / 纹理 | 16 / 27 | 15 / 11 |
| 非黑像素比例 | 34.5% | 72.4% |
| 600 ms 帧差均值 | 2.97 | 4.98 |

桌面纹理数包含空闲预热的 T6；移动端没有任何相邻场景预热标记。两端深链接都只出现 `scene-load:T7`，没有错误加载 T0。
