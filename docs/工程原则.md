# 工程原则 — Universe Knowledge 平台

> 本文件约束所有 apps/ 下项目的工程行为。各应用有自己的局部规范，但遇冲突以本文件为准。

---

## 一、核心信条

**我们在做的东西值得认真对待。**

这个平台承载着真实的知识，服务着真实的读者。代码的质量直接影响内容的可信度和体验的品质。我们不接受"差不多能用"——每一行代码都应该对得起它所承载的内容。

---

## 二、铁律（违反则驳回）

### 2.1 聚焦原则
每次修改必须收敛到一个明确的问题。禁止在 bug fix 里夹带样式漂移、无关重构或大范围格式化。

### 2.2 不写防御性废代码
- 框架和内部逻辑能保证的地方，不要再 try/catch。
- 只在系统边界（用户输入、外部 API、文件读取）做校验。
- 不写为了"以防万一"的 fallback——如果需要 fallback，说明设计有问题，先修设计。

### 2.3 小文件原则
- 单文件实质代码 > 300 行：必须考虑拆分。
- 函数 > 50 行：必须拆分或证明拆分会降低清晰度。
- 一个函数做一件事。

### 2.4 命名胜于注释
- 好的命名让代码自解释，不需要注释说"这是做什么的"。
- 注释只写**为什么**：隐性约束、奇怪的临时方案、特定 bug 的 workaround。
- 禁止多段式 docstring 或大块注释区。

### 2.5 生产代码保护
- `universe-physics/`（旧位置）和 `human-history/website/`（旧 Vite SPA）在对应 Next.js 版本上线验证前**禁止删除**。
- 任何破坏性操作（rm -rf、force push、drop table）需要明确用户授权。

### 2.6 可访问性底线
- 所有动画必须尊重 `prefers-reduced-motion` 媒体查询。
- 降级方案：跳过动画，直接显示最终状态。

## 三、架构边界

| 层级 | 职责 | 禁止 |
|------|------|------|
| `apps/` 各应用 | 页面路由、业务逻辑、应用级状态 | 直接依赖其他 app 内部模块 |
| `packages/ui/` | 无业务逻辑的通用 UI 组件 | 引入任何 app 特有的业务判断 |
| `apps/*/src/data/` | 结构化知识内容与枚举 | 写入 DOM、读写浏览器状态 |
| `apps/*/src/lib/` | 跨页面通用工具（路由、DOM、格式化） | 引入具体页面业务判断 |
| `docs/` | 中文知识库与平台规范文档 | 存放构建产物或运行时数据 |

### 跨应用共享规则
- 共享 UI 组件通过 `@universe/ui` 包，不跨 app 直接引用。
- 共享工具函数（格式化、常量）同样放 packages/，不在 apps 间互相 import。
- 知识内容数据（历史事件、哲学家信息）属于各自应用，不共享。

## 四、性能预算

| 应用 | JS（gzip） | CSS（gzip） | 目标帧率 |
|------|-----------|-----------|---------|
| universe-physics | ≤ 200KB | ≤ 10KB | Desktop 60fps / Mobile 30fps |
| human-history | ≤ 200KB | ≤ 10KB | - |
| philosophy | ≤ 150KB | ≤ 8KB | - |
| portal | ≤ 100KB | ≤ 6KB | - |

**动画性能**：universe-physics 的 3D 场景是性能最敏感区域，任何新增渲染逻辑必须在目标设备上实测帧率。

## 五、质量门槛

每次交付前必须通过：

```
pnpm typecheck   # TypeScript 无错误
pnpm lint        # ESLint 0 warning（--max-warnings 0）
pnpm format:check  # Prettier 格式一致
pnpm test        # Vitest 全部通过（适用于 universe-physics）
pnpm build       # 构建成功，无新增 console.error
```

human-history（Vite 版本）额外要求：
- `npm run build` 通过
- `/atlas` 支持：初始渲染、点击时代、点击主题、详情面板、返回上级、重置、缩放平移

---

## 六、调试与验证流程

1. **先复现，再修改**。对可视化交互问题必须用浏览器验证至少一条完整用户路径。
2. **涉及路由、Canvas、全局事件时**，必须验证页面切换后没有残留监听器或动画循环。
3. **响应式变更**：至少检查 Desktop（1280px）和 Mobile（375px）两个断点。
4. **不可验证的风险**：必须在交付说明中明确写出，不能假装不存在。

---

## 七、Clean Code 约束

- 命名必须描述业务含义，禁止 `data`、`item`、`temp`、`handleStuff` 等弱语义名称。
- 禁止魔法数字。布局常量、半径、缩放阈值必须命名为常量。
- DOM 字符串只用于局部、静态、安全的片段；包含用户或外部输入时必须用节点 API 写入文本，防止 XSS。
- 页面级状态必须集中声明，生命周期明确：初始化 → 事件绑定 → 动画循环 → 清理。
