# 工程原则 — Universe Knowledge 平台

> 本文件约束根目录统一 Next.js 应用和内容目录（content/）的工程行为。遇冲突以本文件为准。

---

## 一、核心信条

**我们在做的东西值得认真对待。**

这个平台承载着真实的知识，服务着真实的读者。代码的质量直接影响内容的可信度和体验的品质。我们不接受"差不多能用"——每一行代码都应该对得起它所承载的内容。

---

## 二、铁律（违反则驳回）

### 2.1 聚焦原则

每次修改必须收敛到一个明确的问题。禁止在 bug fix 里夹带样式漂移、无关重构或大范围格式化。

### 2.2 不写防御性废代码

- 框架和内部逻辑能保证的地方，不要再 try/catch。
- 只在系统边界（用户输入、外部 API、文件读取）做校验。
- 不写为了"以防万一"的 fallback——如果需要 fallback，说明设计有问题，先修设计。

### 2.3 小文件原则

- 单文件实质代码 > 300 行：必须考虑拆分。
- 函数 > 50 行：必须拆分或证明拆分会降低清晰度。
- 一个函数做一件事。

### 2.4 命名胜于注释

- 好的命名让代码自解释，不需要注释说"这是做什么的"。
- 注释只写**为什么**：隐性约束、奇怪的临时方案、特定 bug 的 workaround。
- 禁止多段式 docstring 或大块注释区。

### 2.5 生产代码保护

- `reference/` 下旧源码在迁移已完成后保留供参考，**禁止删除**，且不应参与当前应用测试、构建或 lint。
- 任何破坏性操作（rm -rf、force push、drop table）需要明确用户授权。

### 2.6 可访问性底线

- 所有动画必须尊重 `prefers-reduced-motion` 媒体查询。
- 降级方案：跳过动画，直接显示最终状态。

## 三、架构边界

| 层级                        | 职责                              | 禁止                            |
| --------------------------- | --------------------------------- | ------------------------------- |
| `app/`                      | App Router 路由、布局、API        | 存放大量领域业务逻辑            |
| `components/`               | 全局与跨领域 UI 组件              | 引入某个领域私有数据结构        |
| `lib/`                      | 全局 loader、搜索、图引擎与工具   | 直接写页面 UI                   |
| `subjects/physics/`         | 物理前端代码（组件/场景/shaders） | 直接引用其他领域私有代码        |
| `subjects/history/`         | 历史前端代码（组件/渲染器）       | 直接引用其他领域私有代码        |
| `subjects/philosophy/`      | 哲学前端代码（组件/lib）          | 直接引用其他领域私有代码        |
| `subjects/life-science/`    | 生命科学前端代码（组件/lib）      | 直接引用其他领域私有代码        |
| `subjects/cosmology/`       | 宇宙学前端代码                    | 直接引用其他领域私有代码        |
| `subjects/mathematics/`     | 数学前端代码                      | 直接引用其他领域私有代码        |
| `subjects/economics/`       | 经济学前端代码                    | 直接引用其他领域私有代码        |
| `subjects/psychology/`      | 心理学前端代码                    | 直接引用其他领域私有代码        |
| `subjects/knowledge-graph/` | 知识图谱前端代码                  | 直接引用其他领域私有代码        |
| `content/`                  | 所有知识内容（TS/JS/MD/MDX）      | 包含 DOM 操作或浏览器运行时代码 |
| `reference/`                | 旧源码参考                        | 被当前测试、构建、lint 依赖     |
| `docs/`                     | 中文知识库与平台规范文档          | 存放构建产物或运行时数据        |

### 跨应用共享规则

- 知识内容存放在 `content/` 目录，通过 `@/content/...` 路径导入。
- 各领域前端代码存放在 `subjects/` 下；跨领域共享逻辑应进入 `lib/` 或 `components/`，不要让某个 `subjects/*` 直接依赖另一个领域的私有模块。

## 四、性能预算

| 门禁              | 预算（gzip） | 验证方式                                      |
| ----------------- | ------------ | --------------------------------------------- |
| 全平台共享初始 JS | ≤ 180KB      | `pnpm bundle-check` 读取 Next build manifest  |
| 门户首页 CSS      | ≤ 40KB       | 按路由 manifest 对实际资源去重求和            |
| 领域路由 CSS      | ≤ 40KB       | 单一Tailwind入口 + 页面样式按路由manifest隔离 |
| 通用文章路由 JS   | ≤ 220KB      | Markdown服务端渲染，交互按文章字段进入小岛    |
| 任一单 JS chunk   | ≤ 285KB      | 防止新增未分割的超大依赖                      |

**路由性能**：`scripts/performance/lighthouse-budget.mjs`为门户、通用文章、生命可视化与知识图谱维护独立性能分数、LCP、TBT和CLS基线。基线是回归门禁，不代表长期目标；首页LCP的长期目标仍为2.5秒以下。

**CSS入口**：`app/globals.css`是唯一允许`@import "tailwindcss"`的文件。领域样式通过根入口的`@reference`公开主题token，自身只输出领域变量、组件和动画；人类历史等大型页面样式必须在对应路由入口导入，禁止重新聚合到领域布局。

**渲染策略**：SSG、首次访问SSG、ISR和动态接口的关键契约必须通过`pnpm audit-rendering`读取生产manifest验证。内容随部署更新的通用文章不使用无收益的定时ISR；日期内容和公开确定性API按1小时或24小时重验证；用户档案接口保持`private, no-store`。详见`docs/CI-CD与渲染策略.md`。

**动画性能**：universe-physics 的 3D 场景是性能最敏感区域，任何新增渲染逻辑必须在目标设备上实测帧率（Desktop 60fps / Mobile 30fps）。

## 五、质量门槛

每次交付前必须通过：

```
pnpm typecheck   # TypeScript 无错误
pnpm lint        # ESLint 0 warning（--max-warnings 0）
pnpm test        # Vitest 全部通过
pnpm build       # 构建成功，无新增 console.error
```

## 六、调试与验证流程

1. **先复现，再修改**。对可视化交互问题必须用浏览器验证至少一条完整用户路径。
2. **涉及路由、Canvas、全局事件时**，必须验证页面切换后没有残留监听器或动画循环。
3. **响应式变更**：至少检查 Desktop（1280px）和 Mobile（375px）两个断点。
4. **不可验证的风险**：必须在交付说明中明确写出，不能假装不存在。

## 七、Clean Code 约束

- 命名必须描述业务含义，禁止 `data`、`item`、`temp`、`handleStuff` 等弱语义名称。
- 禁止魔法数字。布局常量、半径、缩放阈值必须命名为常量。
- DOM 字符串只用于局部、静态、安全的片段；包含用户或外部输入时必须用节点 API 写入文本，防止 XSS。
- 页面级状态必须集中声明，生命周期明确：初始化 → 事件绑定 → 动画循环 → 清理。
