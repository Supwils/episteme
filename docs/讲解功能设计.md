# 讲解功能（Audio Narration）设计

> 给每篇文章配一段**语音讲解**：用户点击即可听。核心原则——**讲解不是文章的朗读**。

## 一、核心判断：讲解 ≠ 朗读正文

类比博物馆语音导览：导览不会念展品旁的说明牌，而是讲"你该注意什么""背后的故事""为什么重要"。讲解也一样——它是**单独创作的音频伴读**，和正文互补：

| 留在阅读（文字擅长，**不**讲）            | 进讲解（口语擅长，**只**讲）           |
| ----------------------------------------- | -------------------------------------- |
| 精确数字+单位、公式、表格、代码、参考文献 | 一个粘得住的类比 / 心智模型            |
| 可扫读、可检索、需反复回看的密集内容      | 破除误解的钩子、故事与现场感           |
| 分层图、记录类型表、证明枚举              | "为什么这重要"、跨域的那根线、情感重量 |

一句话：**阅读给精度，讲解给理解与动机。** 这是 `叙事与引用规范.md` 的口语延伸。

## 二、内容模型

- 讲解脚本是**兄弟文件** `content/<domain>/<section>/<slug>.narration.md`：可选 frontmatter（`title`）+ 正文是纯口语脚本（无 markdown 标记，适合 TTS）。
- opt-in：有就显示「听讲解」按钮，没有就不显示。逐篇铺开，不强制全站。
- **加载器排除**：`*.narration.md` 在 `knowledge-domain.ts`/`generic-kb.ts`/`gen-domain`/`check-content` 里都被显式排除，**绝不会被当成文章**进站点或搜索。

## 三、技术架构

```
.narration.md (脚本)
  → pnpm gen-narration → ElevenLabs TTS → public/audio/narration/<hash>.mp3
                                        → lib/narration-manifest.json (key → 音频)
文章页 (DomainArticle, 服务端)
  → getNarration(domain,section,slug) 读脚本 + 查 manifest 拿音频 URL
  → <NarrationButton>（客户端）写入全局 store
全局底栏 <NarrationPlayer>（挂 ClientShell, ssr:false）
  → 有音频播音频；无音频回退浏览器语音朗读同一脚本（标「预览音色」）
```

- **预生成**（非按需）：`scripts/gen-narration-audio.ts`，按 `sha256(voice+model+script)` **hash 增量**——脚本/音色没变就跳过，re-run 很便宜。
- **不进 prebuild**：TTS 花钱且需 API key，是手动创作步骤（`pnpm gen-narration`）。
- **TTS（多提供商，按领域选）**：`PROVIDER_BY_DOMAIN` 决定每个领域用哪家——**哲学用小米 MiMo**（免费、更适合中文），**其余用 ElevenLabs**。因为 hash 是 `voice|model|script`，切换某领域的提供商只会重生成该领域音频，已有条目 hash 不变、自动跳过。
  - **ElevenLabs**：Adrian – Chinese Mandarin Narration（id `agczkAUlHLowaNnL72Cc`，标准普通话；共享库**直接调用**，API key 缺 `add_voice_from_voice_library` 权限故不加入账户）+ `eleven_multilingual_v2`，输出 mp3。Key=`.env` `ElevenLabs_API_Key`。
  - **小米 MiMo TTS v2.5**：POST `api.xiaomimimo.com/v1/chat/completions`（chat 格式：user=风格指令、assistant=要读的文本），`api-key` 头，模型 `mimo-v2.5-tts`，女声默认 `茉莉`（男声 `苏打`/`白桦`，女声 `冰糖`/`茉莉`），返回 base64 wav（`choices[0].message.audio.data`）。Key=`.env` `Xiaomi_API_Key`。
  - env 覆盖：`ELEVENLABS_VOICE_ID`/`ELEVENLABS_MODEL_ID`、`MIMO_VOICE`/`MIMO_MODEL`。
  - ⚠️ MiMo 只给 wav（24kHz 无压缩，~4MB/篇），比 mp3 大很多；规模化时应转 mp3 或迁 Vercel Blob。
- **回退**：音频未生成时用 Web Speech API 朗读脚本（演示用，标注"预览音色"）——同样读的是**讲解脚本**不是正文，体现"讲解≠阅读"。
- **可访问性**：底栏动画走全局 prefers-reduced-motion；功能本身是视障/阅读障碍/通勤场景的可达性收益。

## 四、试点（已落地）

3 篇 computer-science 文章（走通用 `DomainArticle`）：`concepts/abstraction`、`concepts/dns`、`pioneers/alan-turing`。已用 ElevenLabs 生成中文音频。

## 五、待办（规模化 + 质量护栏）

1. **「不重复正文」机器强制**：`check-content` 加一条——`.narration.md` 与正文文本相似度过高则报警。把"讲解不要和阅读重复"从靠自觉变成红线。
2. **脚本标准**：在 `叙事与引用规范.md` 加「讲解脚本」一节（口语化、2–4 分钟、必须有钩子/类比/为什么重要、禁照搬正文）。
3. **AI 起草**：正文 → 抽取要点 → 改写成口语脚本 → 人工复核，规模化创作。
4. **存储**：规模化后音频从 `public/`（git）迁到 **Vercel Blob**（别把大量二进制塞进 git）。
5. **接入更多渲染路径**：目前只接了 `DomainArticle`；philosophy/economics 等 bespoke 文章页各自接 `getNarration` + `<NarrationButton>`。
6. **字幕高亮**：ElevenLabs 可返回时间戳 → 讲解文稿卡拉OK式高亮（可选）。
