# 工程守则

## 核心原则

1. 可读性优先于技巧。代码应让下一位维护者能直接理解意图、数据流和清理边界。
2. 页面、数据、工具函数保持解耦。页面只负责组织交互和视图，数据模块只暴露结构化内容，`lib/` 只放跨页面复用能力。
3. 每次修改都必须收敛到明确问题。禁止夹带无关重构、样式漂移或大范围格式化。
4. 交互功能必须可回退、可清理、可验证。路由切换、Canvas 动画、窗口监听和滚动监听都要有对应清理逻辑。
5. 历史内容与产品体验同等重要。新增内容必须同时考虑事实表达、中文可读性、卡片/面板承载能力和移动端显示。

## 架构边界

| 区域 | 职责 | 禁止事项 |
|------|------|----------|
| `src/pages/` | 页面渲染、页面内状态、页面内事件 | 跨页面共享全局状态、直接修改其他页面 DOM |
| `src/data/` | 结构化历史内容与枚举 | 写入 DOM、读写浏览器状态 |
| `src/lib/` | 路由、DOM、动画偏好等通用能力 | 引入具体页面业务判断 |
| `src/styles/` | 设计令牌、布局、组件样式 | 依赖 JS 临时注入核心布局 |
| `docs/` | 中文知识库与工程规范 | 存放构建产物或运行时数据 |

## Clean Code 约束

- 函数只做一件事，超过 50 行必须拆分或证明拆分会降低清晰度。
- 命名必须描述业务含义，避免 `data`, `item`, `temp`, `handleStuff` 等弱语义名称。
- 页面级状态必须集中声明，生命周期必须清楚：初始化、事件绑定、动画循环、清理。
- 禁止用魔法数字表达业务规则。布局常量、半径、缩放阈值应命名为常量。
- DOM 字符串只用于局部、静态、安全的片段；包含用户或外部输入时必须使用节点 API 写入文本。

## 调试与验证流程

1. 先复现，再修改。对可视化交互问题必须用浏览器验证至少一条完整用户路径。
2. 修改后至少运行 `npm run build`。
3. 涉及路由、Canvas、动画或全局事件时，必须验证页面切换后没有遗留监听或动画循环。
4. 涉及响应式布局时，至少检查桌面宽度和移动宽度。
5. 无法验证的风险必须在交付说明中明确写出。

## 质量门槛

- 构建必须通过。
- 控制台不得出现新增 error。
- `/atlas` 必须支持：初始渲染、点击时代、点击主题、详情面板、返回上级、重置、缩放和平移。
- 对 `prefers-reduced-motion` 用户必须避免强制长动画。
- 新增规范必须同步到 `docs/开发规范.md` 或 `website/AGENTS.md` 中与执行有关的部分。
