## 1. 项目定位

**产品名 (暂定)**: **universe-physics** —— 一座用浏览器交付的「宇宙 + 物理」知识图谱站。

整站分两个顶层板块，独立成线：

1. **Universe / Cosmic (当前唯一焦点)** — 从可见宇宙的整体结构出发，沿尺度由大到小展开：可见宇宙 → 大尺度纤维结构 → 超星系团 (Laniakea、Great Attractor 等) → 星系群/团 → 星系 → 太阳系 → 行星与天体。核心体验是**跨尺度的平滑动画下钻**与**科学准确的位置关系**。
2. **Physics (后置, 暂不开发)** — 宏观到微观的物理知识。架构需要预留，但**当前阶段不要分散精力**。

### 产品灵魂

**让用户被宇宙的尺度本身震撼，再借这股震撼去理解结构。**

整站差异化在两件事：

1. **视觉与动画的工业级打磨** —— 流畅、克制、有节奏感的尺度切换。这是核心壁垒，不要拿"功能更多"换"动画粗糙"。
2. **科学准确的位置与结构** —— 超星系团方向、本星系群拓扑、太阳系距离比例等，都以当下主流科研为准；不准的部分宁可不画，也不要乱画。

任何 UI / 文案 / 交互的决定都要先回答："它有没有在帮用户感受宇宙的尺度与结构？" 如果没有，就砍掉。

## 4. 工程铁律

1. **行业级出品**。设计与代码都要看着像产品，不像 demo。
2. **性能优先**。动画必须丝滑；为渲染策略和数据结构的选择留出性能预算。详见 `docs/develop/05-performance-budget.md`。
3. **解耦**。一个函数/模块一个职责；渲染、数据、动画三类逻辑分层，不互相串味。
4. **小文件**。单文件实质代码 > 300 行就要考虑拆分。组件不堆 props 大杂烩。
5. **干净易读**。命名 > 注释 > 文档；命名能说清就别写注释。注释只用来说**为什么**，不说**做什么**。
6. **不写防御性废代码**。框架/内部能保证的东西不要再 try/catch；边界 (用户输入、外部 API) 才校验。
7. **每次会话结束必须更新 WORKLOG**。这是硬性规则。下一个代理上来要能直接接手。
