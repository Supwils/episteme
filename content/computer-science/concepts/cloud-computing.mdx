# 云计算

2006 年，亚马逊先后发布了两个不起眼的服务：3 月推出 S3（简单存储服务），8 月推出 EC2（弹性计算云）。定价是：存储每 GB 每月 0.15 美元，计算每小时 0.10 美元。

当时没有人意识到，这是一场彻底重构软件行业商业模型的革命的起点。近二十年后，云计算市场年规模已超过 5000 亿美元。

在更窄口径的云基础设施（IaaS 与 PaaS）市场上，三巨头的格局相当稳固：据 Synergy Research 统计，2025 年第一季度全球云基础设施支出约 909 亿美元，其中 AWS 占 29%、微软 Azure 占 22%、谷歌云占 12%，三家合计约 65%。剩下的三分之一才留给阿里云、甲骨文、IBM 等所有其他厂商瓜分。

## 破除误解：云不是"别人的电脑"

"云就是别人的电脑"——这句话流传很广，有一定讽刺意味，但它掩盖了云计算最重要的价值主张。

云计算的本质不是地点的变化（从你的机房到数据中心），而是**运营模式的根本转变**：

- 从**资本支出（CapEx）到运营支出（OpEx）**：不再一次性购买服务器，而是按使用量付费
- 从**固定容量到弹性容量**：峰值期间扩展，低谷时缩减，不为闲置资源付费
- 从**自己运维到托管服务**：操作系统补丁、硬件故障、网络配置由云商负责

这三个转变的叠加，让一个五个人的初创公司从第一天起就能使用和 Google 相同级别的基础设施。

## 一个酝酿了半个世纪的旧梦

另一个常见误解是：云计算是 2006 年随 AWS 一起诞生的新事物。其实它的内核是个老想法。

1961 年，人工智能先驱约翰·麦卡锡（John McCarthy）在麻省理工学院百年校庆的演讲中预言：计算也许有一天会像电话系统一样，被组织成一种"公共事业（public utility）"。在他的设想里，算力可以像水电一样按表计量、按需取用。

1966 年，加拿大学者道格拉斯·帕克希尔（Douglas Parkhill）在《计算机公共事业的挑战》（*The Challenge of the Computer Utility*）一书中，已经相当完整地描述了今天云计算的多数特征：弹性供给、按用量计费、多租户共享同一资源池。

"云计算"这个词本身则更晚。它最早出现在 1996 年康柏公司（Compaq）的一份内部文件里，2006 年 8 月被时任谷歌 CEO 埃里克·施密特（Eric Schmidt）在一场搜索引擎大会上公开使用后才真正流行——恰好和 AWS 推出 EC2 是同一个月。

按订阅付费的商业模式也不是 AWS 首创。1999 年，Salesforce 打着"No Software（没有软件）"的旗号成立，把客户关系管理（CRM，管理客户与销售流程的系统）搬进浏览器、按订阅收费，比 EC2 早了整整七年。它要证明的事很简单：用户真正想要的不是"软件光盘"，而是"软件这项服务"。

所以更准确地说，2006 年不是云计算的发明时刻，而是它在硬件成本、网络带宽与虚拟化技术同时成熟后，终于**便宜到可以普及**的时刻。

## 三层服务模型

云计算服务分为三个主要层次，每一层隐藏不同程度的底层细节：

| 层次 | 全称 | 用户控制范围 | 典型产品 |
|------|------|------------|---------|
| **IaaS** | Infrastructure as a Service | 操作系统以上 | AWS EC2, Azure VM |
| **PaaS** | Platform as a Service | 应用代码 | Heroku, Google App Engine |
| **SaaS** | Software as a Service | 配置与数据 | Gmail, Salesforce, Slack |

**IaaS** 给你一台"虚拟机"，你自己安装操作系统、中间件、数据库。灵活性最高，运维责任也最大。

**PaaS** 给你一个运行环境，你只需要上传代码，平台负责部署、扩展、负载均衡。Heroku 的核心命令就是 `git push heroku main`——代码推上去就自动部署了。

**SaaS** 直接给你成品软件，用户只是用，完全不接触底层。

近年还出现了 **FaaS（Function as a Service）** / **Serverless** 模式：代码以函数为单位上传，只有触发时才运行，按实际执行毫秒计费，空闲时不占资源也不付费。AWS Lambda（2014）开创了这一模式。

## 责任共担：云安全里最被误解的部分

把数据交给云商，很多人下意识以为"安全也一并外包了"。这是个危险的误会。

主流云商都遵循**责任共担模型（Shared Responsibility Model）**：云商负责"云本身的安全"（security *of* the cloud）——机房物理安防、硬件、虚拟化层、底层网络；用户负责"云上的安全"（security *in* the cloud）——访问权限配置、数据加密、防火墙规则、操作系统补丁。

这条分界线会随服务层次移动。用 IaaS 时，给虚拟机打补丁、配置应用全归你管；用 SaaS 时，云商替你扛下绝大部分，你只需管好账号权限和数据本身。层次越高，云商承担得越多——这正好对应上一节的三层模型。

危险之处在于：大多数云上数据泄露，根子不在云商被攻破，而在用户这一侧配置出错。Gartner 在 2019 年有一个被反复引用的判断——到 2025 年为止，99% 的云安全事故都将是**客户自己的过错**，最常见的就是把存储桶设成了公开可读、或给账号开了过大的权限。

换句话说，云没让安全消失，只是把责任重新切了一刀，而很多人没意识到自己分到了哪一半。

## 弹性与自动扩展

云计算最重要的能力之一是**自动扩展（Auto Scaling）**。

以电商大促为例：日常流量可能需要 10 台服务器，双十一峰值可能需要 10000 台。传统模式要么长期准备 10000 台（99% 时间闲置），要么峰值时崩溃。

云计算的解决方案是：根据监控指标（CPU 占用率、请求队列长度）自动增加或减少实例数量。峰值时启动几百台额外实例，只需几分钟，峰值结束后自动释放。

这种能力依赖于底层的**虚拟化技术**——物理服务器可以运行数十个虚拟机，云商通过动态调度，从自己的大规模资源池中为客户分配资源。

## 多云与混合云

**多云（Multi-Cloud）**：同时使用多家云商的服务，避免供应商锁定，提高容灾能力。挑战在于不同云商的 API 不同，数据迁移成本高。

**混合云（Hybrid Cloud）**：自建数据中心与公有云共存，通过专线或 VPN 互联。适用于有合规要求（数据不能出境）或已有大量本地基础设施的企业。

**边缘计算（Edge Computing）**：把计算推到网络边缘（靠近用户的节点），减少延迟。AWS Wavelength、Azure Edge Zones 是典型产品。对于自动驾驶、工厂自动化等对延迟极敏感的场景，云中心太远，必须在本地处理。

## 云的定价模型

云计算的计费单位精细到毫秒（FaaS）或秒（部分 IaaS），主要模式：

- **按需计费（On-Demand）**：随用随付，最贵但最灵活
- **预留实例（Reserved Instances）**：提前承诺使用一年或三年，价格降低 30-60%
- **竞价实例（Spot Instances）**：使用云商的闲置资源，价格可低至按需价格的 10%，但可能被随时回收

精明的云成本管理（FinOps）已成为专业领域：自动识别闲置资源、选择最优购买方式、实施资源使用上限。

## 微服务与容器化：云的架构范式

云计算不只改变了基础设施，也推动了软件架构的转型：

**微服务架构（Microservices）**：把一个大型应用拆分成多个小的、独立部署的服务，每个服务只负责一个功能（如用户服务、订单服务、支付服务）。Martin Fowler 和 James Lewis 在 2014 年系统总结了这一模式。

微服务的好处：不同服务可以用不同语言开发，可以独立扩展（只扩容流量大的服务），故障隔离（一个服务崩溃不影响其他服务）。挑战：分布式系统的所有问题（网络延迟、服务发现、分布式事务）全部浮出水面。

**Docker（2013）**：容器化技术的普及化里程碑。容器把应用和它的所有依赖（运行时、库、配置）打包成一个可移植的镜像，"在我机器上能跑"的问题消失了。容器比虚拟机更轻量（共享操作系统内核，启动只需毫秒）。

**Kubernetes（K8s，2014，Google 开源）**：容器编排系统，管理数千个容器的部署、扩缩容、故障自动恢复。起源于 Google 内部使用了十年的 Borg 系统。成为云原生应用的事实标准基础设施，被 AWS、Azure、GCP 全部支持。

三者的关系：微服务提供架构思路，Docker 提供打包标准，Kubernetes 提供运行时管理。这三者共同构成了"云原生（Cloud Native）"的技术基础。

## 代价与争议

**供应商锁定（Vendor Lock-in）**：深度使用某家云商的特有服务（如 AWS DynamoDB、Lambda）后，迁移到另一家的成本极高。Kubernetes 和 Terraform 等开源工具试图提供"云无关"的抽象层，但效果有限。

锁定还有一个更隐蔽的经济机制：**数据出口费（egress fee）**。数据传入云几乎免费，传出却要按量收钱——以 AWS 为例，入站全部免费，出站到公网每月前 100GB 免费、之后约每 GB 0.09 美元起阶梯计价。数据越多，搬走的成本越高，这种"进得来、出不去"的不对称被业界称为"数据引力（data gravity）"。监管已经开始介入：欧盟《数据法案》（Data Act，2025 年 9 月生效）规定，自 2027 年 1 月起云商不得就更换供应商收取切换费与数据出口费；谷歌、微软已先行宣布对部分跨云数据传输免费或仅按成本价收取。

**安全与数据主权**：把数据存在他人的数据中心，信任链变长。2018 年生效的欧盟 GDPR、2021 年生效的中国《数据安全法》都对数据存储的地理位置提出了要求，导致跨国企业必须在不同地区部署独立的云基础设施。

**成本失控**：云计算的弹性是双刃剑——如果不加监控，资源使用量可能在不知不觉中暴涨。有大量案例是初创公司因为一个 bug 引发的无限循环导致月账单从 100 美元暴涨到数十万美元。

**"上云一定更便宜"也是个误解。** 对流量起伏剧烈、增长不确定的业务，弹性确实省钱；但对负载稳定、长期接近满载的业务，长租云资源未必比自建划算。一个被广泛讨论的反例是开发出 Ruby on Rails 框架的 37signals 公司：其联合创始人 DHH 公开表示，公司 AWS 年账单一度高达约 320 万美元，于是花约 70 万美元买下戴尔服务器、把业务迁回自建机房，迁移后年成本降到约 130 万美元；存储部分也从 AWS 每年约 150 万美元换成不到 20 万美元的本地方案，预计五年共节省约 1000 万美元，而这一切由一个十人团队完成、未增一名新人手。这股"下云（cloud repatriation）"思潮提醒人们：云的真正价值是弹性与免运维，而不是无条件的低价。

**环境影响**：全球数据中心耗电量约占全球用电量的 1-2%。大型云商在积极投资可再生能源，但随着 AI 训练需求暴增，数据中心能耗预计大幅上升。国际能源署（IEA）2024 年报告估计，全球数据中心用电量到 2026 年可能翻倍。

## 跨域连接

- **经济学**：云计算把 IT 基础设施从固定成本转为可变成本，降低了创业门槛，对科技行业的竞争格局产生了深远影响；而麦卡锡半个世纪前"计算即公共事业"的设想，本质上是把算力当作电网一样的基础设施来定价。
- **法学与监管**：数据出口费、跨境数据流动与供应商锁定，正在成为反垄断与数字监管的新战场，欧盟《数据法案》是其中代表性的立法。
- **[[distributed-systems]]**：云计算的核心技术基础是分布式系统，见分布式系统条目。
- **[[virtualization]]**：虚拟化是云计算的底层实现机制，见虚拟化条目。
- **哲学**：数据主权和云计算的关系触及"谁拥有你的数据"这一更深的哲学问题，与隐私权理论密切相关。
