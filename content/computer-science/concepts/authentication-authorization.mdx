# 身份验证与授权

每次你登录一个网站，后台都在运行两个截然不同的问题：

**你是谁？**（身份验证，Authentication，简称 AuthN）

**你能做什么？**（授权，Authorization，简称 AuthZ）

这两个问题的区分至关重要，但在日常语言中经常被混用。一个正确回答了"你是谁"的系统，仍然可能在"你能做什么"上出错——比如，一个已验证身份的用户可以访问其他用户的私人数据，这是授权失败，不是身份验证失败。OWASP（开放式 Web 应用安全项目）的年度十大漏洞报告中，"Broken Access Control"（访问控制失效，即授权问题）常年名列榜首。

## 破除误解：密码不是最佳身份验证

密码是数字世界最普遍的身份验证机制，也是最脆弱的一个。问题不在于密码的概念，而在于人类使用密码的方式：

- 在多个网站复用相同密码（一处泄露，处处危险）
- 使用易猜的密码（"123456"每年都是最常用密码）
- 把密码写在便利贴上（绕过所有技术防护）

更根本的问题是：密码是一种"你知道什么"（something you know）的验证方式，只有一个因子。

**多因素认证（Multi-Factor Authentication，MFA）** 要求同时提供至少两种不同类型的凭据：

| 类型 | 示例 |
|------|------|
| 你知道什么（Knowledge） | 密码、PIN |
| 你拥有什么（Possession） | 手机、硬件安全密钥（YubiKey） |
| 你是什么（Inherence） | 指纹、人脸识别 |

Google 内部研究（2019）显示，强制所有员工使用硬件安全密钥（FIDO2 标准）后，账户钓鱼攻击降至零。

## 身份验证的技术实现

### 密码哈希与加盐

正确的密码存储方式是**绝不存储明文密码**，存储其密码学哈希。但直接哈希仍有漏洞：攻击者可以预先计算所有常见密码的哈希（彩虹表，Rainbow Table）。

解决方案是**加盐（Salting）**：为每个密码生成唯一的随机字符串（盐），与密码组合后再哈希。这使得彩虹表攻击完全失效。

推荐的密码哈希算法不是 SHA-256（速度太快，可以暴力破解），而是专门设计为慢速的：
- **bcrypt**（1999）：内置成本因子，可以随硬件提升调整计算量
- **Argon2**（2015）：密码哈希竞赛冠军，抗 GPU 并行攻击，OWASP 推荐首选

### 会话管理与 Cookie

HTTP 协议是无状态的——每个请求独立，服务器不记得之前的请求。如何维持登录状态？

**会话（Session）**：登录成功后，服务器生成一个随机的会话 ID，返回给客户端（通常通过 Cookie），客户端后续请求携带此 ID。服务器在内存或数据库中维护会话状态。

**JWT（JSON Web Token）**：把用户信息和权限直接编码到令牌中（经过服务器私钥签名），客户端持有令牌，服务器无需查询数据库即可验证。适合分布式系统。注意：JWT 令牌一旦签发，在过期前无法无效化（除非维护黑名单）——这是与有状态会话的根本权衡。

## OAuth 2.0：授权的工业标准

你是否使用过"用 Google 账号登录"或"用 GitHub 账号登录"？这背后是 **OAuth 2.0** 协议（2012）。

OAuth 解决的问题：**在不共享密码的情况下，让第三方应用访问你的资源**。

四个角色：
- **资源所有者（Resource Owner）**：你（用户）
- **客户端（Client）**：第三方应用（如某个使用 GitHub 登录的网站）
- **授权服务器（Authorization Server）**：GitHub 的 OAuth 服务器
- **资源服务器（Resource Server）**：GitHub 的 API

授权码流程：
1. 你在第三方应用点击"用 GitHub 登录"
2. 跳转到 GitHub 的授权页面，GitHub 询问你是否允许该应用访问你的某些信息
3. 你同意后，GitHub 返回授权码（Authorization Code）给第三方应用
4. 第三方应用用授权码换取访问令牌（Access Token）
5. 第三方应用用令牌调用 GitHub API，获取你的用户信息

关键点：第三方应用**始终不知道你的 GitHub 密码**。你可以随时在 GitHub 设置中撤销该应用的访问权限。

**OpenID Connect（OIDC）**：建立在 OAuth 2.0 之上，专门用于身份认证（OAuth 本身只定义了授权）。OIDC 在 OAuth 的访问令牌之外，还提供了 ID Token（包含用户身份信息的 JWT），成为"用 xxx 账号登录"功能的标准实现。

## 授权模型

**基于角色的访问控制（RBAC，Role-Based Access Control）**：把权限分配给角色，把角色分配给用户。例：管理员角色有所有权限，普通用户只有读权限。简单，适合用户角色明确的系统。

**基于属性的访问控制（ABAC，Attribute-Based Access Control）**：根据多种属性（用户属性、资源属性、环境属性）动态决定访问权限。例："销售部门的员工，在工作时间，可以访问当前季度的销售数据"。灵活，适合复杂场景，但实现和维护更复杂。

**基于策略的访问控制（PBAC）**：用声明式的策略语言（如 AWS IAM Policy、Open Policy Agent）描述访问控制规则，集中管理。

## 无密码认证的未来

**FIDO2 / WebAuthn（2019，W3C 标准）**：用公钥密码学实现无密码登录。设备（手机、笔记本）生成密钥对，私钥存在设备的安全芯片（TEE/Secure Enclave）中，公钥注册到服务器。登录时用指纹或 PIN 解锁设备，设备用私钥签名服务器的挑战，无需传输密码。

这种方式抗钓鱼攻击（私钥从不离开设备，且绑定到特定域名，假网站无法获取签名）、抗数据库泄露（服务器只存公钥，泄露无法使用）。苹果 Passkeys、Google Passkeys 正在基于 WebAuthn 推动密码的消失。

## 代价与争议

**遗忘账户的困境**：无密码认证依赖于设备，丢失设备可能永久失去账户访问权——除非有备用验证方式。如何平衡安全性和账户恢复能力，仍是开放问题。

**OAuth 的滥用**：OAuth 被广泛用作"使用第三方账号登录"，但它设计目标是授权，不是认证。许多早期实现没有正确验证令牌的发放对象（`state` 参数），导致 CSRF 漏洞。

**生物特征的不可撤销性**：密码泄露后可以更换，但指纹和人脸泄露后无法更换。这使得生物特征的存储安全比密码更重要，也是为什么 WebAuthn 在设备本地处理生物特征而不是上传服务器。
